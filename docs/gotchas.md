# Gotchas and migration notes

Surprising-but-intentional behaviors worth knowing about, collected in one
place. Each links to the full documentation.

## Templates

- **A template without the `.tmpl` suffix is copied byte-for-byte.** If
  your custom template contains `{{ ... }}` and comes out unrendered, the
  file is missing the `.tmpl` suffix. Conversely, files that use
  `{{ ... }}` for their own purposes (GitHub Actions workflows) must *not*
  have the suffix. See
  [Rendered vs verbatim files](custom-templates.md#rendered-vs-verbatim-files).
- **`foo.yml` and `foo.yml.tmpl` side by side is an error**, because both
  would produce the same output file.
- **`metadata.json` and `jig.toml` cannot be templated.** They are always
  generated by jig itself; template trees containing them get a warning
  and the files are ignored. See
  [The module template tree](custom-templates.md#the-module-template-tree).
- **Private https template repositories are not supported yet** — use ssh
  (via ssh-agent) for private repos. See
  [Supported transports](custom-templates.md#supported-transports-and-authentication).
- **A changed ssh host key always fails, with no override.** Only
  *unknown* hosts can be auto-accepted with `--ssh-accept-new`. If a
  server legitimately rotated its key, run `ssh-keygen -R <host>` and
  connect again. See
  [Host key verification](custom-templates.md#host-key-verification).
- **`ssh_accept_new` is never read from `jig.toml` or `metadata.json`.**
  Trust decisions only come from your own config, environment, or flags —
  a cloned repository must not be able to change security behavior for
  the people who clone it. See
  [jig.toml](jig-toml.md#trust-decisions-are-never-read-from-jigtoml).

## `renew`

- **`jig renew` does nothing on a fresh module.** The `[renew]` allowlist
  in `jig.toml` is empty by default; nothing is overwritten until the
  module opts in. See [the allowlist](commands/renew.md#the-allowlist).
- **A successful remote renew rewrites `jig.toml`**, updating the recorded
  commit — hand-written comments in the file do not survive. See
  [jig renew](commands/renew.md#template-source).

## `build`

- **Ignore files are not consulted.** `.pdkignore` and `.pmtignore` are
  ignored (and warned about); packaging is controlled by the spec
  allowlist plus the `[build]` section of `jig.toml`. `.gitignore` is
  left alone — it belongs to git, not the build. See
  [What gets packaged](commands/build.md#what-gets-packaged).
- **`pkg/`, `.git/`, `jig.toml`, and `.gitkeep` markers are never
  packaged**, in both `deny` and `allow` modes.

## Ruby-backed commands

- **`jig validate`, `jig test unit`, and `jig msync` need a Ruby
  toolchain** (`bundle` plus the module's gems) — or a container engine
  with the voxbox runner enabled. See
  [Running through voxbox](voxbox.md).

## Scaffolding

- **Component names must not include the module name prefix.** `jig new
  class bar` inside module `foo` creates `foo::bar`; passing `foo::bar`
  is wrong. Namespaced names like `bar::baz` are fine.
- **`--force` never deletes an existing module directory outright** — it
  creates a timestamped backup first.

## Migrating from jig 1.x

See [Upgrading to 2.0](upgrading-to-2.0.md) for the full guide, including a
per-module checklist. The highlights:

- **`jig update` is now [`jig msync`](commands/msync.md)**: run
  `jig msync update`.
- **Template trees need reshaping.** Files used to be rendered
  unconditionally and some had special names (`gitignore` instead of
  `.gitignore`). Add the `.tmpl` suffix to rendered templates and move
  files to their literal output paths. See
  [the migration note](custom-templates.md#the-module-template-tree).
- **Template settings moved out of `metadata.json`.** `template-url`,
  `template-ref`, and `template-commit` keys in `metadata.json` are
  ignored with a warning; move them to the `[template]` section of
  [`jig.toml`](jig-toml.md#template) and remove them from
  `metadata.json`.

## Migrating from PDK

- [`jig convert`](commands/convert.md) overwrites `Gemfile`, `Rakefile`,
  and `spec/spec_helper.rb` with jig's versions, which is usually enough
  to run `jig validate` and `jig test unit` on a PDK-generated module.
- [`jig validate`](commands/validate.md) mirrors `pdk validate` (syntax,
  puppet-lint, rubocop).
- Leftover `.pdkignore` files are ignored by
  [`jig build`](commands/build.md); it warns and suggests removing them.
  The equivalent workflow is `action = "allow"` with `exceptions` in the
  `[build]` section of [`jig.toml`](jig-toml.md#build).
