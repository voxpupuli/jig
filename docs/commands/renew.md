# `jig renew`

Re-renders the module's template-managed files from the latest templates
and overwrites them, so a template change can be rolled out across many
modules without hand-editing each one.

```
jig renew [--dry-run]
```

## The allowlist

Only files matching the `[renew]` paths allowlist in the module's
[`jig.toml`](../jig-toml.md) are touched. The allowlist is empty by
default, so nothing can be overwritten until the module opts in. Patterns
are gitignore-style globs matched against paths relative to the module
root:

```toml
[renew]
paths = [".github/**", "Gemfile", "Rakefile", "spec/spec_helper.rb"]
```

Allowlisted files whose rendered content differs are overwritten, files
already matching the template output are left alone, and allowlisted files
the module does not have yet (added to the template tree after
scaffolding) are created. `metadata.json` and `jig.toml` are generated by
jig and are never renewed.

With `--dry-run` nothing is written; each file that would change is
reported with a diff.

## Template source

The template source is resolved the same way as for
[`jig new`](new.md#template-source-flags) (it accepts the same
`--template-dir`, `--template-url`, `--template-ref`, and
`--ssh-accept-new` flags): flags first, then the `[template]` section of
`jig.toml` — re-fetching the latest commit of the recorded ref — then
`template_dir` from the [config file](../configuration.md).

After a successful renew from a remote repository, the `[template]`
section of `jig.toml` is updated to the commit that was fetched. Note that
this rewrites `jig.toml`, so hand-written comments in it do not survive.
