# Upgrading to 2.0

jig 2.0 introduces the per-module [`jig.toml`](jig-toml.md) config file,
reworks how templates are laid out and rendered, changes how `jig build`
selects files, and renames a command. Most modules keep working without
changes, but custom template directories and modules that used
`.pdkignore`-style files or recorded template settings in `metadata.json`
need attention.

## Quick checklist

Per module:

- [ ] Run `jig new module` once somewhere, or add a
      [`jig.toml`](jig-toml.md) by hand, so the module has one to carry
      its settings.
- [ ] Move `template-url` / `template-ref` / `template-commit` out of
      `metadata.json` into the `[template]` section of `jig.toml`.
- [ ] Replace `.pdkignore` / `.pmtignore` with a `[build]` section in
      `jig.toml`, then delete the ignore files.
- [ ] Replace `jig update` with `jig msync update` in scripts and CI.
- [ ] Optionally opt in to [`jig renew`](commands/renew.md) by adding a
      `[renew]` paths allowlist.

Per custom template directory:

- [ ] Add the `.tmpl` suffix to every template that should be rendered.
- [ ] Move files with special names to their literal output paths.

Each item is explained below.

## New: per-module `jig.toml`

Settings that belong to a module rather than to a user now live in a
[`jig.toml`](jig-toml.md) in the module root, created by
`jig new module` and committed with the module. It holds the `[template]`
source, the `[renew]` allowlist, and the `[build]` packaging rules. All
sections are optional; existing 1.x modules without a `jig.toml` simply
get jig's defaults.

## Template settings moved out of `metadata.json`

jig 1.x recorded the remote template source as `template-url`,
`template-ref`, and `template-commit` keys in `metadata.json`. Those keys
are no longer supported: jig ignores them and warns — both when
resolving the template source and during the metadata validation that
runs for `jig build` and `jig release`.

Move them to the `[template]` section of `jig.toml` and remove them from
`metadata.json`:

```toml
[template]
url    = "ssh://git@my.git.server/jig_templates.git"
ref    = "main"
commit = "<commit the templates were fetched at>"
```

See [Remote template repositories](custom-templates.md#remote-template-repositories).

## Build: `.pdkignore` and `.pmtignore` are no longer consulted

`jig build` no longer reads ignore files. By default it now packages
**only** the files the
[Puppet module specification](https://github.com/puppetlabs/puppet-specifications/pull/157)
allows in a published module (`manifests/`, `lib/`, `data/`,
`metadata.json`, ...). Development files like `Gemfile`, `spec/`, and
dotfiles stay out with no configuration at all — for most modules this
means you can simply delete `.pdkignore`/`.pmtignore` and change nothing
else. The build warns about any leftover ignore file and suggests
removing it (`.gitignore` is exempt — it belongs to git).

If the module ships files outside the spec allowlist, or you want the
old denylist behavior back, add a `[build]` section to `jig.toml`:

```toml
# Extend the spec allowlist with extra files (recommended):
[build]
action     = "deny"
exceptions = ["/mycustomfile.txt"]
```

```toml
# Or mimic 1.x/PDK: package everything except the exceptions — this is
# the old .pdkignore denylist relocated into jig.toml:
[build]
action     = "allow"
exceptions = ["/spec/**", "/Gemfile"]
```

See [What gets packaged](commands/build.md#what-gets-packaged).

## Templates are now literal-path trees

The template layout and rendering rules changed; custom template
directories from 1.x will likely need updating.

- **Only files ending in `.tmpl` are rendered** (with Go's
  text/template); the suffix is stripped in the output
  (`README.md.tmpl` → `README.md`). In 1.x every template was rendered
  unconditionally. A 1.x template carried over without the suffix is now
  copied verbatim, with `{{ ... }}` left unrendered — that's the
  symptom to look for.
- **Files live at their literal output paths.** Special names are gone:
  `gitignore` is now `.gitignore`, `spec/init_spec.rb` is now
  `spec/classes/init_spec.rb`, and so on. The `module/` tree mirrors the
  generated module exactly, so you can also add arbitrary files jig
  knows nothing about.
- **`metadata.json` and `jig.toml` are reserved.** They are always
  generated by jig itself; template trees containing them get a warning
  and the files are ignored.

To migrate a custom template directory, run
[`jig templates dump`](commands/templates.md#jig-templates-dump) to see
the new layout, rename your rendered templates to add `.tmpl`, and move
specially-named files to their literal paths. Use
[`jig templates resolve`](commands/templates.md#jig-templates-resolve)
to confirm each template resolves from where you expect.

See [Custom templates](custom-templates.md) for the full layout.

## `jig update` is now `jig msync`

The `update` command was renamed to reduce confusion about what it does:
it runs [modulesync](https://github.com/voxpupuli/modulesync), not a
jig-native update. Replace `jig update` with `jig msync update` in
scripts and CI. `jig msync` is a general passthrough, so other msync
subcommands work too. See [`jig msync`](commands/msync.md).

For jig-native, template-driven refreshes of managed files, use the new
`jig renew` instead (below).

## New commands in 2.0

- [`jig renew`](commands/renew.md) — re-renders template-managed files
  from the latest templates and overwrites them, so a template change
  can be rolled out across many modules. Opt-in per module: only files
  matching the `[renew]` paths allowlist in `jig.toml` are touched, and
  the allowlist is empty by default. `--dry-run` shows a diff without
  writing.
- [`jig templates resolve`](commands/templates.md#jig-templates-resolve)
  — shows which source a template name resolves from and every path
  checked, for debugging custom template setups.
